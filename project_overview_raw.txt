⚙️  Principe directeur de cette phase
Chaque chantier produit du code TypeScript pur (0 React), testable en isolation avec Vitest.
L'objectif est qu'ajouter une zone, un monstre ou un type de quête ne nécessite
  qu'une entrée JSON — zéro modification de code applicatif.
Les modules communiquent exclusivement via le GameEventBus.
  Aucun engine ne connaît l'existence d'un autre engine.
Chantier 1 — Architecture & Event Bus
Découplage des modules · Data loader typé · Feature flags · Couche storage abstraite
1.1  Problème actuel & objectif
Sans bus d'événements, les modules se couplent directement : le combat engine appelle le quest engine qui appelle le achievement engine. Chaque nouvelle feature crée de nouvelles dépendances. L'objectif est qu'un module émette un événement sans savoir qui l'écoute.
🔗  Avant vs Après
AVANT (couplage direct) :
  combatEngine.killMonster() → questEngine.onMonsterKilled() → achievementEngine.check()
  → Ajouter les succès oblige à modifier combat-engine.ts
APRÈS (event bus) :
  combatEngine.killMonster() → bus.emit('MONSTER_KILLED', payload)
  questEngine.on('MONSTER_KILLED', ...)   // écoute indépendante
  achievementEngine.on('MONSTER_KILLED', ...)  // écoute indépendante
  statsEngine.on('MONSTER_KILLED', ...)   // ajouté sans toucher au combat

export type GameEventMap = {
  // Combat
  MONSTER_KILLED:    { monsterId: string; zoneId: string; loot: ItemDrop[]; xpGained: number }
  PLAYER_DAMAGED:    { damage: number; source: string; newHp: number }
  PLAYER_DIED:       { zoneId: string; killedBy: string }
  ATTACK_DODGED:     { attackerId: string }
  COMBAT_STARTED:    { zoneId: string; monsterId: string }
  COMBAT_ENDED:      { zoneId: string; victory: boolean }
  // Métiers
  SKILL_ACTION_DONE: { skillId: SkillId; actionId: string; xpGained: number; outputs: ItemDrop[] }
  SKILL_LEVEL_UP:    { skillId: SkillId; newLevel: number }
  SKILL_MILESTONE:   { skillId: SkillId; level: number; bonus: string }
  // Craft
  CRAFT_COMPLETED:   { recipeId: string; skillId: SkillId; output: ItemDrop; xpGained: number }
  CRAFT_FAILED:      { recipeId: string; reason: string }
  UPGRADE_BUILT:     { upgradeId: string; level: number }
  // Inventaire
  ITEM_ADDED:        { itemId: string; quantity: number; source: string }
  ITEM_REMOVED:      { itemId: string; quantity: number }
  ITEM_EQUIPPED:     { itemId: string; slot: SlotId }
  INVENTORY_FULL:    { itemId: string; quantityLost: number }
  // Agriculture
  PLANT_SOWED:       { plotIndex: number; seedId: string; readyAt: number }
  PLANT_READY:       { plotIndex: number; seedId: string }
  PLANT_HARVESTED:   { plotIndex: number; seedId: string; outputs: ItemDrop[]; perfect: boolean }
  // Économie
  ITEM_SOLD:         { itemId: string; quantity: number; goldGained: number }
  ITEM_BOUGHT:       { itemId: string; quantity: number; goldSpent: number }
  GOLD_CHANGED:      { delta: number; newTotal: number; source: string }
  // Quêtes
  QUEST_STARTED:     { questId: string }
  QUEST_OBJECTIVE:   { questId: string; objectiveIndex: number; current: number; target: number }
  QUEST_COMPLETED:   { questId: string; rewards: QuestReward }
  DAILY_RESET:       { date: string; newQuestIds: string[] }
  // Session
  GAME_LOADED:       { isGuest: boolean; offlineDuration: number }
  OFFLINE_RECOVERED: { duration: number; summary: OfflineSummary }
  SAVE_COMPLETED:    { target: 'local' | 'cloud' }
}
type Listener<T> = (payload: T) => void
class GameEventBus {
  private listeners = new Map<string, Set<Listener<any>>>()
  on<K extends keyof GameEventMap>(event: K, listener: Listener<GameEventMap[K]>): () => void {
    if (!this.listeners.has(event)) this.listeners.set(event, new Set())
    this.listeners.get(event)!.add(listener)
    return () => this.listeners.get(event)?.delete(listener)  // unsubscribe fn
  }
  emit<K extends keyof GameEventMap>(event: K, payload: GameEventMap[K]): void {
    this.listeners.get(event)?.forEach(fn => {
      try { fn(payload) }
      catch (err) { console.error(`[EventBus] Error in ${event} listener:`, err) }
    })
  }
  // Vider tous les listeners (utile pour les tests)
  clear(): void { this.listeners.clear() }
}
export const bus = new GameEventBus()  // singleton partagé
1.3  Data Loader — Accesseurs typés
Plutôt que d'importer les JSON directement dans chaque fichier, un loader centralisé valide les données au démarrage, expose des accesseurs typés, et lève des erreurs explicites si une référence est invalide (ex: une recette qui pointe vers un itemId inexistant).
engine/data-loader.ts — Loader centralisé
import rawItems    from '@/data/items.json'
import rawSkills   from '@/data/skills.json'
import rawRecipes  from '@/data/recipes.json'
import rawZones    from '@/data/zones_monsters.json'
import rawQuests   from '@/data/quests.json'
// Maps indexées pour lookups O(1)
const itemMap    = new Map(rawItems.items.map(i => [i.id, i]))
const skillMap   = new Map(rawSkills.skills.map(s => [s.id, s]))
const recipeMap  = new Map(rawRecipes.recipes.map(r => [r.id, r]))
const monsterMap = new Map(rawZones.monsters.map(m => [m.id, m]))
const zoneMap    = new Map(rawZones.zones.map(z => [z.id, z]))
const questMap   = new Map([
  ...rawQuests.mainQuests.map(q => [q.id, q]),
  ...Object.values(rawQuests.skillQuests).flat().map(q => [q.id, q]),
])
// Accesseurs avec erreur explicite
export const GameData = {
  item    : (id: string) => { const v = itemMap.get(id);    if (!v) throw new Error(`Unknown item: ${id}`);    return v },
  skill   : (id: string) => { const v = skillMap.get(id);   if (!v) throw new Error(`Unknown skill: ${id}`);   return v },
  recipe  : (id: string) => { const v = recipeMap.get(id);  if (!v) throw new Error(`Unknown recipe: ${id}`);  return v },
  monster : (id: string) => { const v = monsterMap.get(id); if (!v) throw new Error(`Unknown monster: ${id}`); return v },
  zone    : (id: string) => { const v = zoneMap.get(id);    if (!v) throw new Error(`Unknown zone: ${id}`);    return v },
  quest   : (id: string) => { const v = questMap.get(id);   if (!v) throw new Error(`Unknown quest: ${id}`);   return v },
  // Collections filtrées
  itemsOfCategory : (cat: string) => rawItems.items.filter(i => i.category === cat),
  recipesForSkill : (skillId: string) => rawRecipes.recipes.filter(r => r.skill === skillId),
  monstersInZone  : (zoneId: string) => rawZones.monsters.filter(m => m.zone === zoneId),
  actionsForSkill : (skillId: string) => GameData.skill(skillId).actions ?? [],
} as const
// Validation au démarrage (appelée dans app/layout.tsx)
export function validateGameData(): void {
  const errors: string[] = []
  // Vérifier que chaque output de recette pointe vers un item existant
  rawRecipes.recipes.forEach(r => {
    r.inputs?.forEach(i => { if (!itemMap.has(i.itemId)) errors.push(`Recipe ${r.id}: unknown input ${i.itemId}`) })
    if (r.output && !itemMap.has(r.output.itemId)) errors.push(`Recipe ${r.id}: unknown output ${r.output.itemId}`)
  })
  // Vérifier que chaque drop de monstre pointe vers un item existant
  rawZones.monsters.forEach(m => {
    m.drops?.forEach(d => { if (!itemMap.has(d.itemId)) errors.push(`Monster ${m.id}: unknown drop ${d.itemId}`) })
  })
  if (errors.length > 0) throw new Error(`Data validation failed:\n${errors.join('\n')}`) 
}

2.1  Garde-fous dans le moteur offline
Le calcul offline est le système le plus fragile. Un seul NaN peut corrompre toute une sauvegarde. Trois niveaux de protection sont nécessaires.
engine/offline-engine.ts — Guards complets
// NIVEAU 1 : Validation de l'entrée
function validateDelta(rawDelta: number): number {
  if (!Number.isFinite(rawDelta) || rawDelta < 0) {
    console.error('[Offline] Invalid delta:', rawDelta)
    return 0
  }
  const cap = FLAGS.ENABLE_PRAYER_SKILL  // exemple flag qui change le cap
    ? OFFLINE_CAP_GRIMOIRE_H * 3600 * 1000
    : OFFLINE_CAP_DEFAULT_H  * 3600 * 1000
  return Math.min(rawDelta, cap)
}
// NIVEAU 2 : Validation de chaque résultat intermédiaire
function safeXp(base: number, count: number): number {
  const result = base * count
  if (!Number.isFinite(result) || result < 0) return 0
  return Math.floor(result)
}
// NIVEAU 3 : Validation de l'inventaire avant application
function applyOfflineLoot(store: GameStore, loot: Record<string, number>): string[] {
  const warnings: string[] = []
  for (const [itemId, qty] of Object.entries(loot)) {
    if (!Number.isFinite(qty) || qty <= 0) { warnings.push(`Invalid qty for ${itemId}`); continue }
    const current = store.inventory[itemId] ?? 0
    const item    = GameData.item(itemId)
    const capped  = Math.min(qty, item.stackMax - current)
    if (capped < qty) warnings.push(`Stack overflow: ${itemId} (lost ${qty - capped})`)
    if (capped > 0) store.inventory[itemId] = current + capped
  }
  return warnings  // affichés dans l'écran de récupération
}

2.3  Système de migration des saves
Chaque save contient un champ version. Quand SAVE_VERSION augmente, la fonction migrateSave() transforme les saves plus anciennes de façon chaînée jusqu'à la version courante.
lib/save-migrations.ts — Migrations chaînées
export const SAVE_VERSION = 3
type MigrationFn = (state: Record<string, any>) => Record<string, any>
const migrations: Record<number, MigrationFn> = {
  // v1 → v2 : ajout du système d'upgrades
  1: (s) => ({ ...s, upgrades: {}, unlockedFlags: [] }),
  // v2 → v3 : combat décomposé en 8 skills
  2: (s) => ({
    ...s,
    skills: {
      ...s.skills,
      // Migrer l'ancien skill 'combat' vers 'attack' + stats dérivées
      attack:       { level: s.skills.combat?.level ?? 1, xp: s.skills.combat?.xp ?? 0, activeAction: null, actionStartedAt: null },
      strength:     { level: 1, xp: 0, activeAction: null, actionStartedAt: null },
      ranged:       { level: 1, xp: 0, activeAction: null, actionStartedAt: null },
      magic:        { level: 1, xp: 0, activeAction: null, actionStartedAt: null },
      defense:      { level: 1, xp: 0, activeAction: null, actionStartedAt: null },
      dodge:        { level: 1, xp: 0, activeAction: null, actionStartedAt: null },
      constitution: { level: s.skills.combat?.level ?? 1, xp: 0, activeAction: null, actionStartedAt: null },
      prayer:       { level: 1, xp: 0, activeAction: null, actionStartedAt: null },
    },
  }),
}
export function migrateSave(raw: Record<string, any>): GameState {
  let state = structuredClone(raw)  // ne jamais muter l'original
  const startVersion = state.version ?? 1
  for (let v = startVersion; v < SAVE_VERSION; v++) {
    const migrate = migrations[v]
    if (!migrate) throw new Error(`No migration from v${v} to v${v + 1}`)
    state = { ...migrate(state), version: v + 1 }
    console.info(`[Save] Migrated from v${v} to v${v + 1}`)
  }
  return state as GameState
}
2.4  Flux Guest → Register (sans perte)
🔐  Séquence de migration Guest → Compte
1. DÉTECTION    Le joueur clique 'Créer un compte'
   → Lire la save guest depuis localStorage
   → Vérifier que la save est valide (migrateSave si version < courante)
2. AUTH         Supabase signUp(email, password)
   → En cas d'erreur (email déjà pris) → afficher message, save guest INTACTE
3. CONFLIT      Vérifier si une save cloud existe déjà pour ce userId
   → Si oui : modal 'Garder local (X niveaux) ou Cloud (Y niveaux) ?'
   → Si non : passer directement à l'étape 4
4. UPLOAD       saveGameState(newUserId, guestSave)
   → En cas d'erreur réseau : RETRY 3x avec backoff exponentiel
   → En cas d'échec total : save guest conservée, message d'erreur
5. NETTOYAGE    Si upload réussi :
   → localStorage.removeItem('idle-realms-guest-save')
   → localStorage.removeItem('idle-realms-guest-id')
   → localStorage.setItem('idle-realms-user-id', newUserId)
   → bus.emit('GAME_LOADED', { isGuest: false, offlineDuration: 0 })

3.1  Architecture du Quest Engine
Le quest engine n'appelle jamais les autres engines. Il écoute exclusivement les événements du bus et met à jour le store des quêtes en conséquence. Ajouter un nouveau type de quête ne nécessite qu'une entrée JSON + un handler dans la map.
engine/quest-engine.ts — Structure principale
import { bus } from './event-bus'
import { GameData } from './data-loader'
// Map des handlers par type d'objectif
type ObjectiveHandler = (
  objective: QuestObjective,
  event: any,
  current: number
) => number  // retourne la nouvelle valeur de progression
const OBJECTIVE_HANDLERS: Record<string, ObjectiveHandler> = {
  kill: (obj, event: GameEventMap['MONSTER_KILLED'], current) => {
    if (obj.monsterId && obj.monsterId !== event.monsterId) return current
    if (obj.zoneId   && obj.zoneId   !== event.zoneId)   return current
    return current + 1
  },
  gather: (obj, event: GameEventMap['SKILL_ACTION_DONE'], current) => {
    if (obj.skillId && obj.skillId !== event.skillId) return current
    if (obj.actionId && obj.actionId !== event.actionId) return current
    return current + 1
  },
  craft: (obj, event: GameEventMap['CRAFT_COMPLETED'], current) => {
    if (obj.recipeId && obj.recipeId !== event.recipeId) return current
    if (obj.skillId  && obj.skillId  !== event.skillId)  return current
    return current + event.output.qty
  },
  harvest: (obj, event: GameEventMap['PLANT_HARVESTED'], current) => current + 1,
  sell: (obj, event: GameEventMap['ITEM_SOLD'], current) => {
    if (obj.itemId && obj.itemId !== event.itemId) return current
    return current + event.goldGained
  },
  deliver: (obj, event: GameEventMap['ITEM_REMOVED'], current) => {
    if (event.itemId !== obj.itemId) return current
    return current + event.quantity
  },
  reach_level: (obj, event: GameEventMap['SKILL_LEVEL_UP'], current) => {
    if (event.skillId !== obj.skillId) return current
    return event.newLevel  // la cible est un niveau, pas un compteur
  },
  kill_boss: (obj, event: GameEventMap['MONSTER_KILLED'], current) => {
    const monster = GameData.monster(event.monsterId)
    if (monster.rarity !== 'boss') return current
    if (obj.zoneId && obj.zoneId !== event.zoneId) return current
    return current + 1
  },
}
engine/quest-engine.ts — Initialisation et listeners
export function initQuestEngine(store: QuestStore): void {
  // Listener générique — routing vers le bon handler
  function handleEvent(eventType: string, payload: any): void {
    const activeQuests = store.getActiveQuests()
    for (const quest of activeQuests) {
      let questUpdated = false
      quest.objectives.forEach((obj, idx) => {
        if (obj.type !== eventType) return
        if (obj.completed) return
        const handler = OBJECTIVE_HANDLERS[obj.type]
        if (!handler) return
        const current  = quest.progress[idx] ?? 0
        const newValue = handler(obj, payload, current)
        if (newValue !== current) {
          store.updateObjectiveProgress(quest.id, idx, Math.min(newValue, obj.qty))
          bus.emit('QUEST_OBJECTIVE', { questId: quest.id, objectiveIndex: idx,
            current: Math.min(newValue, obj.qty), target: obj.qty })
          questUpdated = true
        }
      })
      // Vérifier si la quête est complète
      if (questUpdated && store.isQuestComplete(quest.id)) {
        completeQuest(quest.id, store)
      }
    }
  }
  // Brancher les événements pertinents sur le handler générique
  bus.on('MONSTER_KILLED',    p => { handleEvent('kill', p); handleEvent('kill_boss', p) })
  bus.on('SKILL_ACTION_DONE', p => handleEvent('gather', p))
  bus.on('CRAFT_COMPLETED',   p => handleEvent('craft', p))
  bus.on('PLANT_HARVESTED',   p => handleEvent('harvest', p))
  bus.on('ITEM_SOLD',         p => handleEvent('sell', p))
  bus.on('ITEM_REMOVED',      p => handleEvent('deliver', p))
  bus.on('SKILL_LEVEL_UP',    p => handleEvent('reach_level', p))
}
function completeQuest(questId: string, store: QuestStore): void {
  const quest   = GameData.quest(questId)
  store.markCompleted(questId)
  // Créditer les récompenses
  applyQuestRewards(quest.rewards, store)
  // Débloquer les quêtes suivantes
  quest.unlocks?.forEach(id => store.unlockQuest(id))
  bus.emit('QUEST_COMPLETED', { questId, rewards: quest.rewards })
}

// Seed unique par joueur par jour
function getDailySeed(playerId: string): number {
  const dayIndex = Math.floor(Date.now() / 86_400_000)  // UTC
  return dayIndex ^ hashString(playerId)
}
function getWeeklySeed(playerId: string): number {
  const weekIndex = Math.floor(Date.now() / 604_800_000)
  return weekIndex ^ hashString(playerId)
}
// Hash FNV-1a simple (déterministe, pas crypto)
function hashString(str: string): number {
  let hash = 2166136261
  for (const c of str) hash = Math.imul(hash ^ c.charCodeAt(0), 16777619)
  return hash >>> 0  // unsigned 32-bit
}
export function generateDailyQuests(
  playerId: string,
  playerSkills: Record<SkillId, { level: number }>,
  count = 3
): string[] {
  const rng  = mulberry32(getDailySeed(playerId))
  const pool = rawQuests.dailyQuestPool.filter(q =>
    isQuestAccessible(q, playerSkills)
  )
  // Sélection sans remise
  const selected: string[] = []
  const available = [...pool]
  while (selected.length < count && available.length > 0) {
    const idx = Math.floor(rng() * available.length)
    selected.push(available[idx].id)
    available.splice(idx, 1)
  }
  return selected
}
export function generateWeeklyQuest(playerId: string): string {
  const rng = mulberry32(getWeeklySeed(playerId))
  const idx = Math.floor(rng() * rawQuests.weeklyQuests.length)
  return rawQuests.weeklyQuests[idx].id
}
// Vérifier si une quête est accessible selon les niveaux du joueur
function isQuestAccessible(quest: any, skills: Record<string, { level: number }>): boolean {
  if (!quest.reqSkills) return true
  return Object.entries(quest.reqSkills).every(
    ([skillId, minLevel]) => (skills[skillId]?.level ?? 1) >= (minLevel as number)
  )
}
3.3  Reset journalier
Le reset des journalières est déclenché côté client au premier chargement du jeu si la date UTC a changé depuis la dernière sauvegarde. Pas de cron serveur nécessaire.
engine/quest-engine.ts — checkDailyReset()
export function checkDailyReset(store: QuestStore, playerId: string): void {
  const todayStr = new Date().toISOString().slice(0, 10)  // 'YYYY-MM-DD'
  if (store.lastDailyDate === todayStr) return  // déjà resetté aujourd'hui
  const newQuestIds = generateDailyQuests(playerId, store.playerSkills)
  store.setDailyQuests(newQuestIds, todayStr)
  // Reset du streak si un jour a été manqué
  const yesterday = new Date(Date.now() - 86_400_000).toISOString().slice(0, 10)
  if (store.lastDailyDate !== yesterday) {
    store.resetStreak()
  } else {
    store.incrementStreak()
  }
  bus.emit('DAILY_RESET', { date: todayStr, newQuestIds })
}

2. Progression Hors-Ligne
Un pilier fondamental : le joueur ne doit pas être pénalisé par l'absence. Le monde continue de tourner pendant qu'il est déconnecté. À son retour, il retrouve les fruits de son absence, calculés précisément.
2.1  Principes Généraux
⏱️  Règles du Offline Progress
Toute action active au moment de la déconnexion continue de s'exécuter hors-ligne.
Le calcul reprend exactement là où il s'était arrêté : même action, même compteur XP.
Plafond hors-ligne : 48h maximum de progression accumulée (évite le burn-out de rattrapage).
Passé 48h, le jeu entre en mode « Veille » : pas de progression, mais conservation de l'état.
Les ressources produites hors-ligne s'accumulent dans un inventaire tampon dédié (pas de perte si inventaire plein).
Les combats hors-ligne se poursuivent jusqu'à la mort du personnage, puis s'arrêtent automatiquement.
2.2  Calcul Offline
À la reconnexion, le serveur (ou le localStorage timestampé) calcule la progression de la manière suivante :
Variable
Formule
Exemple
Temps écoulé (T)
Date reconnexion − Date déconnexion
8h = 28 800 secondes

min(T, 48h)

Le RNG des drops hors-ligne utilise un seed déterministe basé sur l'horodatage de déconnexion. Cela garantit que la progression est reproductible et ne peut pas être manipulée par changement d'heure système.
2.3  Écran de Récupération
À chaque reconnexion après une absence significative (> 10 minutes), un écran de récupération s'affiche. Il résume la progression offline de manière satisfaisante et engageante.

3. Présence Active & Mini-Jeux
Être en ligne ne doit pas être une contrainte, mais un avantage. Les mini-jeux offrent des bonus significatifs au joueur actif, sans jamais pénaliser celui qui joue de façon passive.
3.1  Philosophie des Bonus Actifs
✅  Ce que les mini-jeux FONT
Offrent des bonus temporaires significatifs
Donnent des ressources rares introuvables offline
Créent de la variété dans l'expérience
Récompensent la régularité (connexion quotidienne)
Déblocable progressivement selon niveaux
❌  Ce que les mini-jeux NE FONT PAS
Bloquer la progression offline
Être obligatoires pour avancer
Punir le joueur absent
Devenir un travail répétitif pénible
Nécessiter des réflexes parfaits
3.2  Catalogue des Mini-Jeux
🎣  Mini-Jeu de Pêche — « La Grande Prise »
Accessible depuis la zone de pêche. Disponibilité : 1 partie gratuite toutes les 2h, jusqu'à 3 stockées.
Mécanique — La Grande Prise
Une jauge de tension s'affiche — une ligne de pêche qui vibre selon l'activité du poisson.
Le joueur clique/appuie au bon moment pour « ferrer » : timing dans une fenêtre de 1–2 secondes.
Difficulté : 3 niveaux selon le niveau de Pêche (Easy / Normal / Hard).
Succès : Poisson rare ou épique (introuvable en offline), bonus XP x3 pour la session suivante.
Échec : Poisson commun quand même (pas de punition) + retry dans 30 min.
Maîtrise : Niveau 50 Pêche → fenêtre de timing élargie. Niveau 80 → 2e essai gratuit.
⛏️  Mini-Jeu de Minage — « La Veine Cachée »
Apparaît aléatoirement pendant une session active de minage (toutes les 10–20 min). C'est un événement contextuel, pas une feature séparée.
Mécanique — La Veine Cachée
Une fissure apparaît dans la mine. Le joueur doit « tapper » (clics rapides) ou résoudre un pattern.
Pattern : 4 cases s'illuminent en séquence, reproduire l'ordre dans les 5 secondes.
Succès : Burst de minerais rares (5–15 unités), chance de pierre précieuse, XP bonus.
Difficulté : Le pattern s'allonge avec le niveau de Minage (2→3→4→5 cases).
Si ignoré : L'événement expire après 45 secondes sans pénalité.
⚔️  Mini-Jeu de Combat — « Coup Critique Parfait »
Lors d'un combat actif, une fenêtre d'opportunité apparaît : une barre de puissance animée.
Mécanique — Coup Critique Parfait
Une jauge se remplit et se vide en boucle (2–3 secondes par cycle).
Cliquer dans la zone « Perfect » (20% central de la jauge) → Coup Critique x3 dégâts.
Cliquer dans la zone « Bien » (40% adjacent) → Critique x1.5.
Ne pas cliquer → Coup normal automatique (pas de pénalité).
Fréquence : Une opportunité toutes les 5–8 frappes selon la vitesse d'attaque.
Amélioration : Talent « Œil de Lynx » → zone Perfect agrandie à 35%.
🌾  Mini-Jeu d'Agriculture — « La Récolte Parfaite »
Quand le joueur est en ligne au moment exact de la maturité d'une plante, un bonus se déclenche.
Mécanique — La Récolte Parfaite
Une plante atteint sa maturité → notification + animation sur la parcelle.
Récolter dans les 5 minutes suivant la maturité → Récolte x2 quantité.
Mini-interaction : Identifier parmi 3 plantes celle qui est prête (visuellement distincte).
Passé 5 min → Récolte normale automatique (la plante ne pourrit pas).
Cumul : Récolter 3 plantes en « Parfait » consécutif → bonus Graine Rare offerte.
🧪  Mini-Jeu d'Alchimie — « L'Instillation »
Lors de la préparation de potions, le joueur peut intervenir pour en augmenter la puissance.
Mécanique — L'Instillation
Pendant les 10 dernières secondes du craft d'une potion, une bulle colorée monte dans un tube.
Cliquer au bon niveau pour « stabiliser » la mixture : fenêtre de timing colorée.
Succès : Potion de qualité Supérieure (effets +25%), chance de créer 2 potions au lieu d'1.
Zones : Verte (réussite normale), Bleue (qualité sup.), Rouge (haut risque — potion Exceptionnelle ou ratée).
Raté : Potion de qualité Normale quand même (jamais de perte de ressources).

5.3  Règles de Design du Craft Croisé
📐  Principes pour équilibrer les recettes croisées
Règle 1 — Minimum 2 métiers : Aucun item Rare ou plus ne doit être craftable avec un seul métier.
Règle 2 — Maximum 6 étapes : Au-delà, la chaîne devient décourageante. Réserver aux légendaires.
Règle 3 — Goulot d'étranglement visible : Une ressource rare identifiable incite à progresser dans son métier source.
Règle 4 — Substitutions possibles : Certains ingrédients ont des alternatives moins efficaces (ex : Lingot d'Or remplaçable par 2x Lingot d'Argent avec -10% stats).
Règle 5 — Traçabilité : L'interface montre toujours quelles ressources manquent et quel métier les produit.
Règle 6 — Équilibre farming / achat : Toute ressource craftable est aussi achetable au marché (plus cher), évitant les blocages.
5.4  Interface du Craft Croisé
L'interface de craft affiche la chaîne de production complète pour les recettes multi-étapes, avec un indicateur de disponibilité pour chaque ingrédient intermédiaire.
🖥️  Affichage Recette Multi-Étapes dans l'UI
Vue « Simple » : Affiche uniquement les ingrédients finaux nécessaires (sans les étapes intermédiaires).
Vue « Détaillée » : Déplie chaque étape avec ses ingrédients, métier requis et niveau nécessaire.
Indicateurs de statut :
  ✅  Vert  — Ingrédient disponible en inventaire
  🟡  Jaune — Ingrédient craftable (ressources disponibles, niveau ok)
  🔴  Rouge — Ingrédient manquant (ressource à collecter, ou niveau insuffisant)
Bouton « Craft Automatique » : Lance automatiquement toutes les étapes intermédiaires si les ressources sont présentes.
Estimation de temps : Affiche le temps total estimé pour compléter toute la chaîne.

6. Combat & Exploration
6.1  Mécanique de Combat
Le combat est automatisé et en temps réel (pas de clic requis). Le joueur choisit une zone d'exploration, équipe son personnage, et observe les résultats. L'optimisation vient du choix des équipements, des potions actives, et de l'ordre de combat.
⚔️ Déroulement d'un Combat
1. Le joueur sélectionne une zone (ex: Forêt des Gobelins)
2. Un monstre aléatoire de la zone est généré
3. Le texte de combat s'affiche ligne par ligne (style MUD/log RPG)
4. Chaque frappe est calculée : dégâts, parade, critique
5. En cas de victoire : drop de ressources, XP Combat
6. En cas de mort : retour au village, pas de malus (objets conservés)
7. Les combats se chainên automatiquement si l'option est activée

1.2  Moteur de Jeu — Delta Timestamp
Le moteur de jeu est entièrement découplé de React. Il fonctionne sur des timestamps purs et peut tourner en dehors du navigateur (calcul offline côté serveur possible à terme).
⚙️  Cycle de Vie du Moteur
1. INIT         → Chargement de l'état depuis localStorage (ou Supabase si connecté)
2. TICK ENGINE  → Appel toutes les X secondes (ou au focus de l'onglet / reconnexion)
                  delta = Math.min(now - lastTick, MAX_OFFLINE_SECONDS)
                  → skill-engine.tick(delta)  : progression des actions actives
                  → combat-engine.tick(delta) : résolution des combats
                  → farming-engine.tick(delta): pousse des plantes
3. TICK UI      → setInterval(250ms) : rafraîchit les barres de progression
                  Ne calcule rien — lit uniquement le store
4. SAVE         → Auto-save localStorage toutes les 30s
                  Sync Supabase toutes les 5 min (ou à la déconnexion)
5. OFFLINE CALC → Au chargement, si lastSave > 60s : calcul du temps écoulé
                  → Affichage écran de récupération

4.1  Moteur Tick — tick.ts
Code structure — tick.ts
const MAX_OFFLINE_MS = 72 * 3600 * 1000  // 72h après Grimoire upgrade
export function tickEngine(store: GameStore, nowMs: number): TickResult {
  const delta = Math.min(nowMs - store.lastTickAt, MAX_OFFLINE_MS)
  const results: TickResult = { xpGained: {}, itemsGained: {}, goldGained: 0 }
  // 1. Métiers actifs
  for (const [skillId, skillState] of Object.entries(store.skills)) {
    if (!skillState.activeAction) continue
    const r = skillEngine.tick(skillId, skillState, delta, store)
    mergeResults(results, r)
  }
  // 2. Combat actif
  if (store.combat.active && store.combat.zoneId) {
    const r = combatEngine.tick(store.combat, delta, store)
    mergeResults(results, r)
  }
  // 3. Agriculture
  const r = farmingEngine.tick(store.farmPlots, delta)
  mergeResults(results, r)
  store.lastTickAt = nowMs
  return results
}
4.2  Calcul XP & Niveaux
La table XP précalculée dans skills.json est utilisée pour des lookups O(1). Les niveaux intermédiaires sont interpolés linéairement.
Utilitaire xp-calc.ts
// XP requis TOTAL pour atteindre le niveau N
export function xpForLevel(n: number): number {
  return XP_TABLE[n] ?? Math.floor(n * 83 + 100 * (1.1 ** n - 1) / 0.1)
}
// Niveau correspondant à une quantité d'XP totale
export function levelForXp(xp: number): number {
  // Recherche binaire sur XP_TABLE (99 entrées — O(log 99) ≈ 7 ops)
  let lo = 1, hi = 99
  while (lo < hi) {
    const mid = (lo + hi + 1) >> 1
    if (XP_TABLE[mid] <= xp) lo = mid else hi = mid - 1
  }
  return lo
}
// Progression 0.0–1.0 dans le niveau courant
export function xpProgress(xp: number): number {
  const level = levelForXp(xp)
  const current = xp - xpForLevel(level)
  const needed  = xpForLevel(level + 1) - xpForLevel(level)
  return current / needed
}
4.3  Résolution RNG & Drops
Le RNG utilise un générateur pseudo-aléatoire seedé (mulberry32) pour garantir la reproductibilité des calculs offline. Le seed est dérivé du timestamp de déconnexion.
rng.ts — Générateur seedé pour offline
export function mulberry32(seed: number) {
  return function(): number {
    seed |= 0; seed = seed + 0x6D2B79F5 | 0
    let t = Math.imul(seed ^ seed >>> 15, 1 | seed)
    t = t + Math.imul(t ^ t >>> 7, 61 | t) ^ t
    return ((t ^ t >>> 14) >>> 0) / 4294967296
  }
}
export function rollDrops(drops: DropTable[], rng: () => number): ItemDrop[] {
  return drops
    .filter(d => rng() < d.chance)
    .map(d => ({ itemId: d.itemId, qty: d.qty }))
}

1.3  Déroulement d'un Combat (pseudo-code)
combat-engine.ts — tick(deltaMs)
// Appelé par le moteur tick avec le delta temps écoulé
function tickCombat(state: CombatState, deltaMs: number, store: GameStore): CombatResult {
  const results: CombatResult = { loot: {}, xp: 0, goldGained: 0, log: [] }
  let remaining = deltaMs
  while (remaining > 0 && state.active) {
    const monster = state.currentMonster
    const timeToNextPlayerHit  = state.playerHitCooldown
    const timeToNextMonsterHit = state.monsterHitCooldown
    const step = Math.min(remaining, timeToNextPlayerHit, timeToNextMonsterHit)
    state.playerHitCooldown  -= step
    state.monsterHitCooldown -= step
    remaining -= step
    // Tour joueur
    if (state.playerHitCooldown <= 0) {
      const dmg = calcDamage(store.playerStats, monster.stats, state.rng)
      monster.hp -= dmg
      state.playerHitCooldown = store.playerStats.attackSpeed * 1000
      results.log.push({ type: 'player_hit', dmg, crit: dmg > baseDmg * 1.5 })
    }
    // Tour monstre
    if (state.monsterHitCooldown <= 0) {
      const dmg = calcDamage(monster.stats, store.playerStats, state.rng)
      store.playerHp -= dmg
      state.monsterHitCooldown = monster.stats.attackSpeed * 1000
      results.log.push({ type: 'monster_hit', dmg })
    }
    // Monstre mort → récompenses + nouveau monstre ou fin
    if (monster.hp <= 0) {
      results.loot  = mergeDrops(results.loot, rollDrops(monster.drops, state.rng))
      results.xp   += monster.combatXp
      results.goldGained += rollGold(monster.goldDrop, state.rng)
      if (state.autoRestart) spawnNextMonster(state, store)
      else { state.active = false }
    }
    // Joueur mort → stop (items conservés)
    if (store.playerHp <= 0) {
      store.playerHp = Math.floor(store.playerMaxHp * 0.30)
      state.active = false
      results.log.push({ type: 'player_death' })
    }
  }
  return results
}

1.7  Calcul de la Progression Offline
offline-engine.ts — calculateOfflineProgress()
const MAX_OFFLINE_MS  = 48 * 3600 * 1000  // 48h par défaut
const GRIMOIRE_BONUS  = 72 * 3600 * 1000  // 72h si Grimoire crafté
function calculateOfflineProgress(store: GameStore, nowMs: number): OfflineResult {
  const cap     = store.upgrades.offline_cap ? GRIMOIRE_BONUS : MAX_OFFLINE_MS
  const rawDelta = nowMs - store.lastSaveAt
  const delta    = Math.min(rawDelta, cap)       // plafond anti-abus
  const speedMult = 1 + (store.upgrades.offline_speed ?? 0)  // Montre Aventurier
  const effectiveDelta = delta * speedMult
  const result: OfflineResult = { duration: delta, skills: {}, loot: {}, gold: 0 }
  // Seed RNG déterministe basé sur timestamp de déconnexion
  const rng = mulberry32(store.lastSaveAt)
  // Progression des métiers actifs
  for (const [skillId, state] of Object.entries(store.skills)) {
    if (!state.activeAction) continue
    const action  = getAction(skillId, state.activeAction)
    const toolMult = 1 - getToolBonus(store, skillId)
    const duration = action.baseActionTime * toolMult * 1000
    const count    = Math.floor(effectiveDelta / duration)
    result.skills[skillId] = { actionsCount: count, xp: count * action.xpPerAction }
    // Drops avec RNG seedé
    for (let i = 0; i < count; i++) {
      const drops = rollDrops(action.outputs, rng)
      drops.forEach(d => { result.loot[d.itemId] = (result.loot[d.itemId] ?? 0) + d.qty })
    }
  }
  // Combats offline (si actif + zone sélectionnée)
  if (store.combat.active && store.combat.zoneId) {
    const combatResult = simulateCombatsOffline(store, effectiveDelta, rng)
    mergeInto(result.loot, combatResult.loot)
    result.gold += combatResult.gold
  }
  // Agriculture : récoltes arrivées à maturité
  const farmResult = processFarmOffline(store.farmPlots, nowMs)
  mergeInto(result.loot, farmResult.loot)
  return result
}
2. Stratégie d'Authentification
Le modèle retenu est Guest local → Register optionnel. L'objectif est zéro friction au démarrage tout en permettant la sauvegarde cloud persistante.
2.1  Flux Complet
🔐  États d'authentification possibles
ÉTAT 1 — GUEST (nouveau visiteur)
  → Génération d'un guestId UUID v4 dans localStorage
  → Sauvegarde locale UNIQUEMENT (localStorage['idle-realms-save'])
  → Aucune requête Supabase, jeu jouable instantanément
  → Badge 'Jouer en mode invité' visible dans les paramètres
ÉTAT 2 — REGISTER (conversion guest → compte)
  → Modale : email + mot de passe (Supabase Auth)
  → Migration automatique : sauvegarde guest uploadée dans Supabase
  → guestId supprimé, remplacé par userId Supabase
  → Confirmation par email (optionnel en dev, activé en prod)
ÉTAT 3 — LOGGED IN (compte existant + connecté)
  → Chargement depuis Supabase au démarrage
  → Auto-save localStorage toutes les 30s
  → Sync Supabase toutes les 5 min ou à la fermeture (beforeunload)
  → Conflit local vs cloud → plus récent gagne (compare lastSaveAt)
ÉTAT 4 — LOGGED IN (offline / Supabase indisponible)
  → Fallback localStorage automatique
  → Indicateur visuel 'Sauvegarde locale' dans l'UI
  → Sync reprise automatiquement quand la connexion est rétablie
2.2  Logique de Migration Guest → Compte
auth/migrate-guest.ts
export async function migrateGuestToAccount(
  guestSave: GameState,
  newUserId: string
): Promise<void> {
  // 1. Vérifier qu'il n'existe pas déjà une save cloud pour cet utilisateur
  const existing = await supabase
    .from('game_states')
    .select('saved_at')
    .eq('player_id', newUserId)
    .single()
  if (existing.data) {
    // Conflit : demander au joueur lequel garder
    const cloudDate  = new Date(existing.data.saved_at)
    const localDate  = new Date(guestSave.lastSaveAt)
    const keepLocal  = await showMergeConflictModal(cloudDate, localDate)
    if (!keepLocal) return  // garder la save cloud, abandonner la guest
  }
  // 2. Upload la save guest comme save officielle
  const saveWithUserId = { ...guestSave, playerId: newUserId }
  await supabase.from('game_states').upsert({
    player_id:  newUserId,
    state_json: JSON.stringify(saveWithUserId),
    saved_at:   new Date().toISOString(),
    version:    SAVE_VERSION
  })
  // 3. Nettoyer le guest storage
  localStorage.removeItem('idle-realms-guest-id')
  localStorage.setItem('idle-realms-user-id', newUserId)
}
2.3  Versionnement des Sauvegardes
Chaque sauvegarde contient un champ version. Les migrations sont des fonctions pures qui transforment un state v(N) en v(N+1).
lib/save-migrations.ts
const SAVE_VERSION = 3  // Incrémenter à chaque changement de schéma
const migrations: Record<number, (state: any) => any> = {
  1: (s) => ({ ...s, upgrades: {} }),           // v1→v2 : ajout upgrades
  2: (s) => ({ ...s, unlockedFlags: [] }),       // v2→v3 : ajout flags de zone
}
export function migrateSave(raw: any): GameState {
  let state = raw
  while (state.version < SAVE_VERSION) {
    const migrate = migrations[state.version]
    if (!migrate) throw new Error(`No migration from v${state.version}`)
    state = { ...migrate(state), version: state.version + 1 }
  }
  return state as GameState
}
